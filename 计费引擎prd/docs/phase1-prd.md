# 计费引擎 - 一期PRD

## 一、背景

随着公司业务的快速发展和全球化布局，传统的计费方式已难以满足多业务、多场景的复杂需求。为了更灵活地支持后续业务变化，提高计费的准确性和效率，公司决定构建统一的计费引擎系统。

**核心目标**：
- 实现计费规则的标准化和集中管理
- 提高计费系统的扩展性和灵活性，支持快速响应业务需求变更
- 确保多业务系统间计费规则的一致性
- 降低各业务系统的开发和维护成本

**一期重点**：
- 将维度和费用项抽象为公共模块，实现统一管理
- 建立维度配置体系，支持多维度组合
- 构建费用项管理架构，确保费用规则的标准化
- 为后续的计费规则配置和模型管理奠定基础

## 二、业务选项配置

### 2.1 简述

业务选项配置用于维护一级、二级、三级业务层级关系，作为维度和费用项配置时的业务层级选择来源。

### 2.2 基本信息

| 字段名称 | 是否必填 | 描述 | 示例 |
|----------|----------|------|------|
| 业务ID | | 系统生成，引擎中的唯一标识 | `YW202401010001` |
| 业务编码 | 是 | 业务的编码 | `PACKAGE_FULL` |
| 业务层级 | 是 | 业务层级，可选值：一级业务、二级业务、三级业务 | `一级业务` |
| 上级业务ID | 是 | 所属的上级业务ID，一级业务时为空，否则要选择上级业务ID | `YW202401010001` |
| 业务名称 | 是 | 业务的名称，方便识别 | `小包全段` |
| 业务描述 | 否 | 业务的详细描述 | `包含国内转运、空运、清关等全段业务` |
| 启用状态 | 是 | 业务的状态，如"启用"、"禁用" | `启用` |
| 创建时间 | | 业务的创建时间 | `2024-01-01 00:00:00` |
| 更新时间 | | 业务的最后更新时间 | `2024-01-02 00:00:00` |

### 2.3 层级关联规则

- 一级业务：上级业务ID为空
- 二级业务：必须关联一个一级业务
- 三级业务：必须关联一个二级业务

### 2.4 新增逻辑

1. **选择业务层级**：用户需先选择要创建的业务层级（一级业务、二级业务、三级业务）
2. **填写基本信息**：根据所选层级填写相应的基本信息
3. **关联上级业务**：
   - 二级业务：必须从已有的一级业务中选择
   - 三级业务：必须从已有的二级业务中选择
4. **数据验证**：
   - 业务编码：非空，长度不超过50字符，建议使用大写字母、数字和下划线组合，不允许特殊字符
   - 业务名称：非空，长度不超过50字符，不能包含特殊字符
   - 业务描述：长度不超过200字符
5. **保存**：系统验证数据完整性后保存，保存时触发唯一性校验（上级业务 ID + 业务层级 + 业务名称的组合必须唯一）

### 2.5 编辑逻辑

1. **选择业务**：从业务选项列表中选择要编辑的业务
2. **修改信息**：
   - **可修改字段**：业务名称、业务描述、启用状态
   - **不可修改字段**：业务ID、业务编码、业务层级、上级业务ID
3. **数据验证**：
   - 业务名称：非空，长度不超过50字符，不能包含特殊字符
   - 业务描述：长度不超过200字符
   - 启用状态：
     - 若修改为**启用**：下属业务的启用状态不变，需逐个手动启用
     - 若修改为**禁用**：下属业务的启用状态全部自动改为禁用
     - 若是禁用状态，引擎中各页面选项都不能选择该业务。
4. **保存**：系统验证数据完整性后保存，，保存时触发唯一性校验（上级业务 ID + 业务层级 + 业务名称的组合必须唯一）
5. **层级变更限制**：业务层级不允许修改，若需变更层级，需删除后重新创建

### 2.6 删除逻辑

1. **选择业务**：从业务选项列表中选择要删除的业务
2. **删除前检查**：
   - 检查是否存在下级业务：若存在，需先删除下级业务
   - 检查是否被维度/费用项引用：若被引用，不允许删除
3. **确认删除**：用户确认后执行删除操作

## 四、维度配置

### 4.1 简述

维度配置在计费引擎中作为**业务维度的基础定义容器**，主要定位和作用如下：

- **基础信息定义**：维护维度的核心属性，包括编码、名称、描述、业务层级等基本信息
- **业务层级关联**：明确维度所属的一级、二级、三级业务，确保维度与业务场景的对应关系
- **模型配置引用**：作为模型配置时的维度模板，提供基本信息供模型引用
- **集中管理**：通过统一的配置界面管理所有维度实例，确保维度信息的规范和完整
- **变更追踪**：通过创建时间和更新时间字段，记录维度配置的变更历史

维度配置本身不包含校验规则、获取规则等业务规则，这些规则将在模型配置中进行设置，以确保规则与使用场景的紧密结合和变更的可控性。

### 4.2 基本信息

| 字段名称 | 是否必填 | 描述 | 示例 |
|----------|----------|------|------|
| 维度ID | | 系统生成，引擎中的唯一标识，计费模型选择时使用，规则：拼音首字母+8位年月日+4位自增数字 | `WD202401010001` |
| 所属单据 | 是 | 维度所属的业务单据类型，从已注册的单据类型中选择 | `TruckOrder` |
| 维度编码 | 是 | 后台代码，从所属单据的字段编码中选择，唯一标识 | `SUPPLIER` |
| 字段来源 | 是 | 标识字段来自主单据还是子单据，根据维度编码自动带出，只读字段 | `主单据` |
| 维度名称 | 是 | 维度的名称，方便识别 | `供应商` |
| 维度描述 | 否 | 维度的详细描述 | `合作的物流服务供应商，用于区分不同供应商的计费维度` |
| 一级业务 | 是 | 维度所属的一级业务类型 | `小包全段` |
| 二级业务 | 是 | 一级业务下的具体业务类型 | `国内转运` |
| 三级业务 | 是 | 二级业务下的更细粒度业务类型 | `国内转运` |
| 启用状态 | 是 | 维度的状态，如"启用"、"禁用"，若是"禁用"，则计费模型无法选择该维度 | `启用` |
| 创建时间 | | 维度的创建时间 | `2024-01-01 00:00:00` |
| 更新时间 | | 维度的最后更新时间 | `2024-01-02 00:00:00` |

### 4.3 新增的校验逻辑

在创建或修改维度配置时，系统会执行以下校验逻辑：

- **维度编码**：
  - 非空校验：必填字段，不能为空
  - 确保在引擎中的唯一性
  - 长度校验：限制最大长度为50个字符
  - 格式校验：建议使用大写字母、数字和下划线组合，不允许特殊字符

- **维度名称**：
  - 非空校验：必填字段，不能为空
  - 长度校验：限制最大长度为50个字符
  - 格式校验：不能包含特殊字符（如`<>'"&`等）
  - 唯一性校验：所属单据+维度编码+一级业务+二级业务+三级业务一致的数据不允许重复

- **维度描述**：
  - 长度校验：限制最大长度为200个字符

- **业务层级**：
  - 非空校验：一级业务、二级业务、三级业务均为必填
  - 层级关联校验：确保二级业务属于所选的一级业务，三级业务属于所选的二级业务

- **启用状态**：
  - 非空校验：必填字段，只能选择"启用"或"禁用"

### 4.4 编辑逻辑

1. **选择维度**：从维度配置列表中选择要编辑的维度
2. **修改信息**：
   - **可修改字段**：维度名称、维度描述、启用状态
   - **不可修改字段**：维度ID、所属单据、维度编码、一级业务、二级业务、三级业务
3. **数据验证**：
   - 维度名称：非空，长度不超过50字符，不能包含特殊字符
   - 维度描述：长度不超过200字符
   - 启用状态：
     - 若修改为**启用**：不影响已引用该维度的计费模型
     - 若修改为**禁用**：已引用该维度的计费模型继续生效，但新模型无法选择该维度
     - 若是禁用状态，引擎中各页面选项都不能选择该维度。
4. **保存**：系统验证数据完整性后保存，保存时触发唯一性校验（所属单据+维度编码+一级业务+二级业务+三级业务的组合必须唯一）
5. **层级变更限制**：业务层级不允许修改，若需变更层级，需删除后重新创建

### 4.5 导入的校验逻辑

在批量导入维度配置时，系统会执行以下校验逻辑：

- **数据完整性校验**：
  - 确保所有必填字段（所属单据、维度编码、维度名称、一级业务、二级业务、三级业务、启用状态）都有值
  - 检查导入数据的格式是否符合要求

- **唯一性校验**：
  - 检查所属单据+维度编码+一级业务+二级业务+三级业务一致的数据是否存在重复

- **业务规则校验**：
  - 验证所属单据的有效性：确保导入的单据类型在系统中已注册
  - 验证业务层级的有效性：确保导入的业务层级在系统中已存在
  - 验证启用状态的合法性：确保值为"启用"或"禁用"

- **数据量校验**：
  - 限制单次导入的维度数量，避免系统负载过高

- **错误处理**：
  - 提供详细的错误提示，指明具体的错误原因和位置，系统将在导入结果页显示错误信息，并支持下载错误日志（包含行号、字段名、错误原因）
  - 支持部分导入：正确的数据导入成功，错误的数据返回错误信息

### 4.6 维度变更联动规则

维度配置变更后，系统将执行以下联动处理：

- **维度禁用**：
  - 已引用该维度的计费模型继续生效，但新模型无法选择该维度
  - 系统会在禁用前提示用户确认，告知已引用该维度的模型数量

- **维度编码修改**：
  - 维度编码不允许修改
  - 若确需修改，需先删除该维度（确保无模型引用），再重新创建

- **所属单据修改**：
  - 所属单据不允许修改
  - 若确需修改，需先删除该维度（确保无模型引用），再重新创建

- **其他信息修改**：
  - 维度名称、描述、业务层级等信息修改后，已关联的模型/数据自动同步更新
  - 系统会记录变更历史，确保可追溯

## 五、费用项配置

### 5.1 简述

费用项配置在计费引擎中作为**费用项目的基础定义容器**，主要定位和作用如下：

- **基础信息定义**：维护费用项的核心属性，包括编码、名称、英文名称、描述、费用类型、业务层级等基本信息
- **业务层级关联**：明确费用项所属的一级、二级、三级业务，确保费用项与业务场景的对应关系
- **费用类型关联**：明确费用项所属的一级、二级费用类型，确保费用项与费用分类的对应关系
- **模型配置引用**：作为模型配置时的费用项模板，提供基本信息供模型引用
- **集中管理**：通过统一的配置界面管理所有费用项实例，确保费用项信息的规范和完整
- **变更追踪**：通过创建时间和更新时间字段，记录费用项配置的变更历史
- **统一数据源**：作为公司所有系统费用项的唯一来源，其他系统的费用项全部从这里取，确保费用项的一致性和标准化

| 三级业务 | 是 | 二级业务下的更细粒度业务类型 | `国内转运` |

| 创建时间 | | 费用项的创建时间 | `2024-01-01 00:00:00` |
| 更新时间 | | 费用项的最后更新时间 | `2024-01-02 00:00:00` |

### 5.3 新增的校验逻辑

在创建或修改费用项配置时，系统会执行以下校验逻辑：
4. **数据验证**：
   - 业务名称：非空，长度不超过50字符，不能包含特殊字符
   - 业务描述：长度不超过200字符
- **费用项编码**：
  - 非空校验：必填字段，不能为空
  - 确保在引擎中的唯一性
  - 长度校验：限制最大长度为50个字符
  - 格式校验：建议使用大写字母、数字和下划线组合，不允许特殊字符

- **费用项名称**：
  - 非空校验：必填字段，不能为空
  - 长度校验：限制最大长度为50个字符
  - 格式校验：不能包含特殊字符（如`<>'"&`等）
  - 唯一性校验：所属单据+费用项编码+一级费用类型+二级费用类型 的组合数据不允许重复

- **费用项英文名称**：
  - 格式校验：若填写，需为英文/数字/空格组合，最大长度50字符

- **费用项描述**：
  - 长度校验：限制最大长度为200个字符

- **业务层级**：
  - 非空校验：一级业务、二级业务、三级业务均为必填
  - 层级关联校验：确保二级业务属于所选的一级业务，三级业务属于所选的二级业务

- **费用类型**：
  - 非空校验：一级费用类型、二级费用类型均为必填
  - 类型关联校验：确保二级费用类型属于所选的一级费用类型

- **启用状态**：
  - 非空校验：必填字段，只能选择"启用"或"禁用"

### 5.4 编辑逻辑

1. **选择费用项**：从费用项配置列表中选择要编辑的费用项
2. **修改信息**：
   - **可修改字段**：费用项名称、费用项英文名称、费用项描述、启用状态
   - **不可修改字段**：费用项ID、费用项编码、所属单据、一级费用类型、二级费用类型、一级业务、二级业务、三级业务
3. **数据验证**：
   - 费用项名称：非空，长度不超过50字符，不能包含特殊字符
   - 费用项英文名称：若填写，需为英文/数字/空格组合，最大长度50字符
   - 费用项描述：长度不超过200字符
   - 启用状态：
     - 若修改为**启用**：不影响已引用该费用项的计费模型
     - 若修改为**禁用**：已引用该费用项的计费模型继续生效，但新模型无法选择该费用项
     - 若是禁用状态，引擎中各页面选项都不能选择该费用项。
4. **保存**：系统验证数据完整性后保存，保存时触发唯一性校验（所属单据+费用项编码+一级费用类型+二级费用类型的组合必须唯一）
5. **层级变更限制**：业务层级和费用类型层级不允许修改，若需变更层级，需删除后重新创建

### 5.5 导入的校验逻辑

在批量导入费用项配置时，系统会执行以下校验逻辑：

- **数据完整性校验**：
  - 确保所有必填字段（费用项编码、费用项名称、所属单据、一级费用类型、二级费用类型、一级业务、二级业务、三级业务、启用状态）都有值
  - 检查导入数据的格式是否符合要求

- **唯一性校验**：
  - 检查所属单据+费用项编码+一级费用类型+二级费用类型 的组合数据是否存在重复

- **业务规则校验**：
  - 验证所属单据的有效性：确保导入的单据类型在系统中已注册
  - 验证业务层级的有效性：确保导入的业务层级在系统中已存在
  - 验证费用类型的有效性：确保导入的费用类型在系统中已存在
  - 验证启用状态的合法性：确保值为"启用"或"禁用"

- **数据量校验**：
  - 限制单次导入的费用项数量，避免系统负载过高

- **错误处理**：
  - 提供详细的错误提示，指明具体的错误原因和位置，系统将在导入结果页显示错误信息，并支持下载错误日志（包含行号、字段名、错误原因）
  - 支持部分导入：正确的数据导入成功，错误的数据返回错误信息
