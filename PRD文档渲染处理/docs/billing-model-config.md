# 计费模型配置方案

## 一、背景

随着计费引擎的发展，模型配置作为计费规则的核心组成部分，需要更加标准化和集中化管理。为了支持多业务系统的复杂计费需求，确保模型与业务单据的正确绑定，提高模型配置的效率和准确性，特制定本模型配置方案。

## 二、术语定义

| 术语   | 定义 |
| ---- | ------------------------ |
| 模型   | 计费引擎中用于配置报价的容器，包括主模型和子模型 |
| 主模型  | 在特定业务维度组合下唯一存在的模型 |
| 子模型  | 在特定业务维度组合下可配置多个的模型 |
| 维度   | 用于业务筛选和分组的基础元素 |
| 子维度  | 与主维度绑定的维度，用于支持数组字段下的维度配置 |
| 费用项  | 计费规则的具体计算对象 |
| 匹配逻辑 | 用于业务单据匹配报价的规则 |

## 四、业务规则

### 4.1 基本信息规则

1. **主模型唯一性**：在"业务维度组合（一级业务+二级业务+三级业务+关联单据）"下，主模型唯一存在
   - **唯一性校验时机**：新增 / 编辑主模型时，系统实时校验该维度组合下是否已存在主模型
   - **重复时的报错提示**：若存在则报错提示"该业务维度组合下已存在主模型，不可重复创建"
2. **子模型多配置**：在同一"业务维度组合（一级业务+二级业务+三级业务+关联单据）"下，子模型可配置多个
3. **主子模型关系**：主子模型间的运算关系需在独立配置页面中定义，当前默认通用运算规则为"叠加"，具体含义参考下文的【费用项结果输出处理方案】
4. **状态管理**：
   - **报价接收状态**：控制模型是否接收新的报价录入和修改
     - 禁用：停止接收新报价录入 / 已有报价修改，已有报价保留；启用后恢复录入 / 修改
   - **计费参与状态**：控制模型下的报价是否参与计费计算
     - 禁用：模型下所有报价不参与计费计算，启用后恢复（历史计费数据不回溯）
   - **联动规则**：支持独立配置，无强制联动，但需在界面标注状态含义及影响
5. **报价管理**：对报价不能做物理删除和覆盖，统一按逻辑删除来处理

### 4.2 维度配置规则

1. **维度来源限制**：仅可选择已配置的维度
2. **业务维度限制**：只能选择在"业务维度组合（一级业务+二级业务+三级业务+关联单据）"下的维度
3. **多选限制**：至少选择一个维度
4. **已配置范围**：维度下拉列表仅展示"业务维度组合（一级业务 + 二级业务 + 三级业务 + 关联单据）"下已配置的维度，避免跨业务选择
5. **生效时间**：新增维度后实时生效

### 4.3 费用项配置规则

1. **费用项来源限制**：仅可选择已配置的费用项
2. **费用项限制**：只能选择在"业务维度组合（一级业务+二级业务+三级业务+关联单据）"下的费用项
3. **多选限制**：至少选择一个费用项
4. **已配置范围**：费用项下拉列表仅展示"业务维度组合（一级业务 + 二级业务 + 三级业务 + 关联单据）"下已配置的费用项，避免跨业务选择
5. **生效时间**：新增费用项后实时生效

### 4.4 子维度配置规则

1. **主子维度关系**：子维度必须关联到已配置的主维度
2. **唯一性**：同一主维度下子维度名称唯一
3. **可扩展性**：支持多个子维度配置
4. **引用限制**：已被报价引用的子维度不能删除
5. **配置流程**：开始配置子维度 → 选择主维度 → 输入子维度名称 → 输入子维度描述 → 保存子维度配置 → 保存成功则返回子维度列表，保存失败则报错提示并重新输入

### 4.5 相关匹配逻辑规则

**页面维护逻辑**：

1. **维度来源限制**：仅可选择"维度配置"中已关联到当前模型的维度
2. **多选限制**：每条规则至少选择1个维度
3. **序号唯一性**：序号自动生成且唯一，不可手动修改
4. **序号生成规则**：新增规则默认序号为"当前最大序号 + 1"（优先级最低）
5. **优先级规则**：序号越小优先级越高，处理时按序号从小到大依次执行
6. **删除限制**：至少保留1条规则

#### 4.5.1 校验逻辑处理规则

**处理逻辑**：

1. **校验逻辑**：点击弹窗从"维度配置"中已选择的维度进行多选，保存后格式为【维度1；维度2；维度3】
2. **校验规则**：当报价录入时，系统会根据维护的校验逻辑对报价进行校验，按优先级顺序检查报价传参中的维度是否有传值，只要有一个校验逻辑通过，就停止后续校验
3. **校验流程**：开始录入报价 → 获取所有校验逻辑 → 若没有校验逻辑则跳过校验 → 若有校验逻辑则按序号排序 → 取优先级1的校验逻辑 → 检查该逻辑中的所有维度是否已填写 → 若所有维度都已填写则校验通过并停止后续校验 → 若有维度未填写则检查是否还有下一优先级 → 若有则取下一优先级的校验逻辑并继续检查 → 若没有则所有优先级均未通过并报错提示"必填项未填写完整，请检查以下维度：[未填写的维度列表]"
4. **报错提示**：如果所有校验逻辑都未通过，报错提示"必填项未填写完整，请检查以下维度：[未填写的维度列表]"，维度列表按"维度 1、维度 2、维度 3"格式展示
5. **触发时机**：校验逻辑在报价录入时实时校验，报错提示即时展示在操作界面

#### 4.5.2 匹配逻辑处理规则

**处理逻辑**：

1. **匹配逻辑**：点击弹窗从"维度配置"中已选择的维度进行多选，保存后格式为【维度1；维度2；维度3】
2. **匹配规则**：当业务单据传入时，系统会根据维护的匹配逻辑对报价进行匹配，按优先级顺序做兜底逻辑，第一套维度匹配不上就第二套，循环匹配直到匹配到就停止
3. **匹配流程**：业务单据传入 → 获取所有匹配逻辑 → 若没有匹配逻辑则返回空值或报错 → 若有匹配逻辑则按序号排序 → 取优先级1的匹配逻辑 → 按匹配逻辑中的维度筛选报价 → 若有报价匹配则取匹配到的报价并停止后续匹配 → 若没有报价匹配则检查是否还有下一优先级 → 若有则取下一优先级的匹配逻辑并继续筛选 → 若没有则所有优先级均未匹配并返回空值或报错提示"未找到匹配报价，请检查以下维度：[未匹配的维度列表]"
4. **匹配逻辑设计原则**：优先设计精确匹配条件（多维度组合），再设计模糊匹配条件（少维度组合），形成从精确到模糊的兜底策略
5. **报错提示**：如果所有匹配逻辑都未匹配到报价，报错提示"未找到匹配报价，请检查以下维度：[未匹配的维度列表]"，维度列表按"维度 1、维度 2、维度 3"格式展示
6. **触发时机**：匹配逻辑在业务单据传入时实时校验，报错提示即时展示在操作界面

## 六、与其他模块的关系

### 6.1 与业务单据的关系

- 模型配置前需先注册业务单据类型
- 模型绑定到具体业务单据
- 不同单据的模型可以重名，同一单据内的主模型必须唯一

### 6.2 与维度的关系

- 模型引用已配置的维度
- 维度用于业务筛选和分组
- 模型维度变更时，系统需要检查并提示对现有报价的影响

### 6.3 与费用项的关系

- 模型引用已配置的费用项
- 费用项是计费规则的具体计算对象
- 模型费用项变更时，系统需要检查并提示对现有计费规则的影响

### 6.4 与因子的关系

- 因子作为计费规则的基础元素，被费用项引用
- 模型通过费用项间接使用因子进行费用计算
- 因子变更时，系统需要检查并提示对相关费用项和模型的影响

### 6.5 与子维度的关系

- 子维度必须关联到已配置的主维度
- 子维度用于支持数组字段下的维度配置
- 子维度变更时，系统需要检查并提示对相关报价的影响

### 6.6 与报价模块的关系

- **模型与报价的关系**：模型与报价为"一对多"关系，报价关联模型 ID

## 七、费用项结果输出处理方案

### 7.1 核心原则

**引擎输出原则**：引擎负责输出所有匹配到的费用项结果，对于模型内的同名费用项，单独列示不进行合并；对于主子模型之间的同名费用项，默认采用叠加规则，符合条件的就把子模型金额加到主模型对应费用项的金额中，不符合条件就单独列示。

### 7.2 处理规则

1. **费用项来源**：明确费用项的来源，包括模型类型（主模型/子模型）和维度类型（主维度/子维度）
2. **匹配维度**：详细描述费用项匹配的维度信息，包括主维度和子维度
3. **输出顺序**：按费用项来源的优先级排序，优先级从高到低为：主模型-主维度 > 主模型-子维度 > 子模型-主维度 > 子模型-子维度
4. **合并条件**：符合条件的费用项结果需要满足以下条件：费用编码相同，且计算规则相同，且计算因子相同（或者计算因子计算条件相同）
5. **业务系统处理**：业务系统可以根据费用项来源、匹配维度进行后续处理，如合并、去重或展示

## 八、边界情况处理

1. **模型重名处理**：通过"业务维度组合（一级业务+二级业务+三级业务+关联单据）"隔离，允许不同业务维度组合使用相同的模型名称
2. **模型删除保护**：被引用的模型不允许删除，避免影响现有计费规则
3. **模型修改适配规则**：修改模型维度 / 费用项后，系统校验已有报价是否符合新配置，若不符合（如缺少新维度值），则弹窗提示
4. **维度变更**：
   - 允许新增维度：现有模型可以添加新的维度，以适应业务发展需求
   - 禁止删除维度：已关联的维度不能删除，避免影响现有报价的完整性
   - 维度变更影响：添加新维度时，系统应提示用户新维度对现有报价的影响（如需要补充填写新维度的值）
5. **费用项变更**：
   - 允许新增费用项：现有模型可以添加新的费用项，以适应业务发展需求
   - 禁止删除费用项：已关联的费用项不能删除，避免影响现有计费规则的完整性
   - 费用项变更影响：添加新费用项时，系统应提示用户新费用项对现有计费规则的影响
